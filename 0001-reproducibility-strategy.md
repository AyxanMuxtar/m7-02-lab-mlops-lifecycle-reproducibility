# ADR 0001: Reproducibility strategy for NorthStar models

**Status:** Accepted  
**Date:** 2025-05-29  
**Deciders:** ML Platform Team, ML Lead  

---

## Context

NorthStar's ML team currently stores trained models in S3 with names like `eta_v2_FINAL.onnx`. There is no dataset versioning, no experiment tracker, and no pinned training environment — only the model file survives. When a prediction goes wrong in production, the team cannot reproduce the training run that produced it: the dataset that day is gone, the code that week may have diverged on `main`, and the Python environment is whatever happened to be installed on the engineer's laptop. This makes debugging impossible and auditability non-existent.

---

## Decision

We adopt a four-layer reproducibility strategy. Every layer is pinned with a specific tool; "we'll track it somehow" is not acceptable.

### Layer 1 — Environment

**Tool: Docker (training images) + `pip-compile` (lockfile)**

Every training run executes inside `northstar-trainer:<git_sha>`, a Docker image built from `docker/Dockerfile.trainer` and pushed to our ECR registry on every merge to `main`. The image pins the base Python version (`python:3.11.9-slim`), system libraries (via `apt-get` with explicit versions in the Dockerfile), and Python packages via a `requirements.lock` file generated by `pip-compile requirements.in --generate-hashes`. The lock file is committed to the repo. No training run may use a floating `requirements.txt` or an unpinned base image.

The image digest (SHA256) is recorded in the MLflow run lineage as `docker_image_digest`. Rebuilding the same image from the same commit produces a byte-identical result (verified in CI with `docker manifest inspect`).

### Layer 2 — Data

**Tool: DVC (Data Version Control) with S3 remote**

All training datasets are managed by DVC. Datasets are never overwritten in place. Each ingestion job produces a new DVC-tracked file (`dataset_<date>.parquet`) whose content hash (SHA256) is committed to the DVC `.dvc` pointer file in Git. The MLflow lineage field `dataset_hash` holds this DVC content hash. To reproduce a run: `dvc checkout dataset_hash=<hash>` restores the exact bytes.

The frozen evaluation holdout (`eval_set_2024Q4.parquet`) is tagged in DVC as immutable and locked from modification by a CI check that fails if the holdout's DVC hash changes. It will not be regenerated mid-cycle.

### Layer 3 — Code

**Tool: Git (mandatory, SHA pinned in lineage) + MLflow Projects**

Training is launched via `mlflow run . --version <git_sha>`, which clones the repo at that exact commit into a temporary directory and runs `train.py` from there. The `git_sha` is recorded automatically by MLflow in the run's lineage and cross-checked by our CI registration script before any model version is written to the registry. A run whose `git_sha` is not a tag or a commit reachable from `main` is rejected.

Hotfixes to `train.py` that are not committed to the repo cannot produce registered models — the CI gate enforces this.

### Layer 4 — Randomness

**Tool: explicit `--seed` CLI argument + `seed_everything()` from PyTorch Lightning**

Every `train.py` invocation requires a `--seed` integer argument (default: `42`). The script calls `pytorch_lightning.seed_everything(seed, workers=True)`, which seeds Python's `random`, NumPy, PyTorch, and CUDA. The seed value is logged to MLflow as a parameter and stored in the registry lineage field `seed`. DataLoader workers are seeded via `worker_init_fn`.

For hyperparameter search jobs (Optuna), the sampler is seeded with the same seed, and the full trial database (`study_<run_id>.db`) is archived to S3 alongside the model artifact.

---

## Alternatives rejected

- **No dataset versioning — rely on S3 timestamps and object versioning.** S3 timestamps tell you when a file was uploaded, not what it contained, and object versioning is not tracked against training runs. If the ingestion pipeline rewrites the same key (as it currently does), the old data is silently gone. Rejected: unrecoverable data loss risk.

- **Conda environments instead of Docker.** Conda environment YAML files do not guarantee identical binary packages across platforms or over time because many packages are pulled from channels that mutate (package builds change, older versions are yanked). Docker layer digests are content-addressed and immutable once pushed to ECR. Rejected: insufficient reproducibility guarantee.

- **MLflow auto-logging only, no explicit lineage fields.** MLflow's autolog captures model parameters and some metrics but does not enforce that `dataset_hash`, `docker_image_digest`, or `git_sha` are present. Without mandatory fields, lineage is incomplete by the time someone needs it (during an incident). Rejected: passive logging does not produce the audit trail we need.

---

## Consequences

- **Every engineer runs training inside Docker.** Local `python train.py` without Docker is allowed for development iteration but cannot produce a registered model. This adds a ~3-minute image pull to onboarding and requires Docker daemon access in CI.

- **Dataset storage costs increase.** DVC with an immutable-append policy means we never delete old dataset versions. At current ingestion volume (~2 GB/week for ETA, ~500 MB/month for dispatcher), S3 costs grow by roughly $15/month. Accepted as the cost of auditability.

- **The frozen evaluation holdout must be managed carefully.** Any change to `eval_set_2024Q4.parquet` breaks the CI check and requires a deliberate, team-approved ADR to roll over to a new holdout. This is intentional: changing the holdout silently would invalidate all historical metric comparisons.

---

## Revisit if

A regulatory requirement (e.g. EU AI Act Article 13 conformity assessment, or a large logistics customer's vendor audit) demands byte-level model provenance beyond what MLflow lineage + DVC provide — at that point, evaluate a dedicated ML metadata store such as Marquez or a commercial alternative (Weights & Biases Artifacts with access controls).
